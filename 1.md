Ты — ведущий fullstack-разработчик и архитектор систем автоматизированной издательской верстки.

Твоя задача — спроектировать и реализовать production-ready MVP приложения AVGST Catalog Builder: системы, которая автоматически или полуавтоматически собирает профессиональные PDF-каталоги проектов домов AVGST.

Не ограничивайся описанием архитектуры. Изучи существующий репозиторий, составь короткий план и приступай к реализации. После каждого этапа запускай проверки. Не создавай фиктивные данные там, где можно получить реальные данные с avgst.ru.

==================================================
1. ЦЕЛЬ ПРОЕКТА
==================================================

Пользователь должен иметь возможность сформировать каталог, указав:

- типы домов;
- конкретные проекты;
- количество проектов;
- порядок сортировки;
- наличие или отсутствие цен;
- дату актуальности;
- заголовок каталога;
- подзаголовок;
- тип обложки;
- формат документа;
- версию для экрана или печати;
- правила выбора композиций;
- контактные данные;
- необходимые служебные страницы.

После настройки система должна:

1. Получить реальные данные проектов с avgst.ru.
2. Загрузить исходные изображения и планировки.
3. Нормализовать характеристики проектов.
4. Проверить комплектность и качество материалов.
5. Выбрать подходящую композицию страницы для каждого проекта.
6. Собрать содержание и разделители.
7. Сверстать каталог.
8. Провести автоматический preflight.
9. Сформировать PDF.
10. Показать журнал ошибок и предупреждений.
11. Сохранить конфигурацию сборки для повторного использования.

Это не генератор новых домов и не нейросетевая перерисовка архитектуры.

Запрещено:

- придумывать проекты;
- изменять архитектуру домов;
- перерисовывать планировки;
- искажать пропорции изображений;
- заменять недостающие материалы сгенерированными изображениями;
- использовать случайную композицию, которую нельзя воспроизвести.

Уникальность страниц должна обеспечиваться детерминированной библиотекой композиций и правилами их выбора.

==================================================
9. ПРАВИЛА РАБОТЫ С ИЗОБРАЖЕНИЯМИ
==================================================

Экстерьеры:

- разрешается crop;
- запрещается distortion;
- должен сохраняться главный объект;
- object-position можно задавать вручную;
- должен поддерживаться safe area.

Планировки, фасады и разрезы:

- всегда показываются целиком;
- не обрезаются;
- не растягиваются;
- располагаются на белом или нейтральном фоне;
- сохраняют исходные пропорции;
- подписи и размеры должны оставаться читаемыми.

Для каждого placement поддержи:

- contain;
- cover;
- object-position;
- padding;
- background;
- min effective DPI.

Добавь возможность вручную настроить crop/position для конкретного asset в конкретной сборке.

10. ПОЛИГРАФИЯ И OUTPUT PROFILES
==================================================

Нужно поддержать два профиля.

SCREEN:

- RGB;
- оптимизированный размер файла;

- изображения без агрессивного сжатия;

- предупреждение ниже 250 DPI;
- ошибка ниже 150 DPI;

- опциональные crop marks;
- встраивание шрифтов;
- подготовка под PDF/X-4 через PrintPdfRenderer;
- возможность указать ICC-профиль;
- отсутствие интерактивных элементов в финальной печатной версии.

Если полноценный PDF/X-4 нельзя гарантированно сформировать Chromium, честно показывай статус:

«PDF подготовлен для проверки, но не сертифицирован как PDF/X-4».

Не маскируй технические ограничения.

==================================================
11. PREFLIGHT
==================================================

Перед финальной сборкой выполни проверки:

- отсутствует основной экстерьер;
- отсутствует планировка;
- неизвестна площадь;
- неизвестны габариты;
- неизвестно количество спален;
- изображение повреждено;
- изображение слишком маленькое;
- недостаточный effective DPI;
- несовместимый цветовой профиль;
- текст не помещается;
- выбранный layout несовместим с проектом;
- отсутствует шрифт;
- отсутствует renderer;
- цена не имеет даты актуальности;
- битая ссылка проекта;
- дублирующиеся assets;
- пустая страница;
- некорректное количество страниц;
- незаполненный дисклеймер.

Результат preflight:

{
  "status": "passed | warning | failed",
  "errors": [],
  "warnings": [],
  "checks": []
}

Сборка для экрана может завершаться с warnings.

Печатная сборка не должна завершаться при критических ошибках.
