Ты — ведущий full-stack-разработчик и архитектор систем автоматизированной верстки. Твоя задача — спроектировать и реализовать рабочее приложение для автоматической и полуавтоматической сборки PDF-каталогов проектов домов компании AVGST.

Сайт компании:
https://avgst.ru/

ГЛАВНАЯ ЦЕЛЬ

Нужно создать не разовый PDF и не скрипт с одним шаблоном, а полноценный publishing pipeline:

1. Получать реальные проекты домов с сайта AVGST.
2. Загружать оригинальные изображения, планировки и характеристики.
3. Позволять пользователю выбирать проекты и параметры каталога.
4. Автоматически выбирать подходящую композицию страницы.
5. Собирать визуально разнообразный, но стилистически единый каталог.
6. Выпускать экранную и печатную версии PDF.
7. Выполнять автоматическую предпечатную и визуальную проверку.
8. Позволять вручную переопределять композицию отдельных страниц.

Никакой генерации или перерисовки архитектуры домов нейросетями быть не должно. Используются только исходные материалы AVGST. Изображения нельзя деформировать, планировки нельзя перерисовывать или менять.

БИБЛИОТЕКА КОМПОЗИЦИЙ

Не делай один шаблон для всех проектов.

Создай минимум 8 layout variants:

1. hero_plan_right
   Большой экстерьер слева, планировка справа.

2. hero_top_plan_bottom
   Панорамный экстерьер сверху, планировка и параметры снизу.

3. split_equal
   Экстерьер и планировка занимают равные части.

4. plan_focus
   Крупная планировка, экстерьер как дополнительный акцент.

5. two_exteriors_plan
   Два экстерьера и одна планировка.

6. vertical_exterior_plan
   Для вертикального или близкого к квадратному рендера.

7. two_floor_plans
   Для двухэтажного дома с двумя планировками.

8. flagship_spread
   Двухстраничный разворот для флагманского проекта.

Каждый шаблон должен быть отдельным Jinja2 partial или компонентом:

templates/layouts/hero_plan_right.html
templates/layouts/hero_top_plan_bottom.html
и т. д.

Каждый layout обязан:

- сохранять пропорции изображений;
- не обрезать планировки;
- не растягивать изображения;
- корректно работать с длинными названиями;
- не допускать выхода текста за пределы;
- поддерживать необязательные поля;
- иметь тестовый fixture.

АВТОМАТИЧЕСКИЙ ВЫБОР КОМПОЗИЦИИ

Создай LayoutSelector.

Он должен детерминированно выбирать шаблон по правилам:

- количество экстерьеров;
- количество планировок;
- этажность;
- пропорции основного изображения;
- площадь дома;
- количество текста;
- статус проекта как флагманского;
- необходимость избежать повторения одного шаблона подряд.

Пример:

- 1 горизонтальный exterior + 1 plan → hero_plan_right;
- очень широкий exterior → hero_top_plan_bottom;
- квадратный exterior → split_equal;
- 2 exterior + 1 plan → two_exteriors_plan;
- 2 plan → two_floor_plans;
- flagship=true → flagship_spread.

Добавь ограничение:

- одинаковый layout не должен использоваться более двух раз подряд;
- соседние страницы должны визуально чередовать положение главного изображения;
- пользовательское override всегда имеет приоритет.

Результат должен быть воспроизводимым: одинаковый набор входных данных и настроек дает одинаковый PDF.

ПРАВИЛА ИЗОБРАЖЕНИЙ

Exterior:

- разрешается crop;
- запрещено изменение пропорций;
- точка кадрирования должна настраиваться;
- поддержать object-position;
- по умолчанию center center.

Floor plan:

- всегда contain;
- никогда не обрезать;
- белый или нейтральный фон;
- не искажать пропорции;
- проверить минимальную читаемость;
- при недостаточном размере выдать warning.

Facade/section:

- всегда contain;
- запрещено обрезать размеры и подписи.

Добавь ручную настройку focal point для каждого изображения.

2. Не удаляй полезный прототип без причины.
3. Если уже существуют Python-скрипты загрузки Tilda или сборки PDF, используй их как источник проверенной логики, но переработай в нормальную модульную архитектуру.
4. Не перезаписывай пользовательские изменения.
5. Проверь наличие AGENTS.md и следуй его требованиям.

ВАЖНО

Не ограничивайся созданием плана или описанием архитектуры. Реализуй работающий MVP.

После каждого этапа:

1. Запускай тесты.
2. Исправляй ошибки.
3. Проверяй приложение.
4. Фиксируй результат в кратком progress report.
5. Продолжай, пока не будет собран реальный PDF минимум из:
   - 10 модульных домов;
   - 10 панельно-каркасных домов.

КРИТЕРИИ ГОТОВНОСТИ

Работа считается выполненной, если:

1. Проекты успешно загружаются с AVGST/Tilda.
2. В интерфейсе отображаются реальные карточки.
3. Можно выбрать 20 проектов.
4. Для страниц автоматически выбираются разные layouts.
5. Планировки не обрезаются и не деформируются.
6. Экстерьеры отображаются без искажения пропорций.
7. Формируется корректное содержание.
8. Генерируется экранный PDF.
9. Создаются previews всех страниц.
10. Формируется preflight-report.
11. Пользователь может изменить layout проекта.
12. После пересборки PDF отражает изменения.
13. Проект запускается через Docker Compose.
14. README позволяет другому разработчику запустить систему с нуля.

Начни с анализа репозитория, затем представь краткий план реализации и сразу переходи к созданию файлов и кода.
